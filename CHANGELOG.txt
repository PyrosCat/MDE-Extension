  FIX — Approach A: explicit return false in all 16 onMessage listeners

  chrome.runtime.sendMessage broadcasts to every loaded extension context.
  Any listener that neither returns true (async response pending) nor
  returns false (synchronous done) leaves Chrome holding an open channel.
  When a callback was registered by the sender, Chrome fires:
    "A listener indicated an asynchronous response by returning true, but
     the message channel closed before a response was received."
  13 listeners were missing explicit return false for unhandled messages.
  3 listeners used bare return; (returns undefined — same problem).
  All 16 listeners (across 15 files) now have explicit return false as the
  final fallthrough statement. Background.js real_myListener was already
  correct and is confirmed by test.
  Files changed: capture.js, chat.js, feed.js, feedback.js, files.js,
  invoice.js, invoiceselect.js, monitor.js, phrases.js, popup.js,
  raterhub.js, social.js, spell.js, task-actions.js.
  (chat.js, invoice.js, invoiceselect.js, and popup.js had additional
  bare return; statements inside handled branches found during final review
  — all converted to return false.)
  16 new tests.

  FIX — Approach B: SendSafeRuntimeMessage fire-and-forget mode

  SendSafeRuntimeMessage (utils.js) previously substituted dummyResponse
  when no callback was provided, passing it to chrome.runtime.sendMessage
  unconditionally. This told Chrome to expect a response for every message
  — including fire-and-forget messages like CONSOLELOG, ALERTFROMBACKGROUND,
  and status updates — which no listener ever responds to.
  Fix: only pass a callback to sendMessage when the caller explicitly
  provides one. With no callback, Chrome doesn't track the channel and
  the "message channel closed" error cannot occur for that call.
  3 new tests.

  ROADMAP — Approach C: targeted messaging (planned for 3.0.0)

  Approaches A and B eliminate the error but the underlying architecture —
  16 listeners receiving every broadcast message — remains. Approach C
  (deferred to 3.0.0 with the ES module refactor) replaces broadcast
  runtime messages with targeted chrome.tabs.sendMessage(specificTabId)
  or long-lived chrome.runtime.Port connections, so messages only reach
  the context that cares. This eliminates the broadcast problem entirely.

  FIX — offscreen.js: createDocument callback/await race + wrong reason

  Two bugs in setupOffscreen caused the "message channel closed" error to
  recur even after the audio.js return-false fix:

  1. chrome.offscreen.createDocument was called with a callback (MV3 only
     supports the Promise API — callbacks are silently dropped). The
     callback was never invoked, so SendSafeTabMessage fired immediately,
     before the offscreen document existed. audio.js had no listener yet,
     FORRHTEMP received no response, and Chrome fired the error.

  2. reasons: ['CLIPBOARD'] — this is intentional. AUDIO_PLAYBACK causes
     Chrome to auto-close the offscreen document 30 seconds after audio
     stops, forcing a new createDocument call for every subsequent alert.
     CLIPBOARD has no lifetime limit, keeping the document alive for
     repeated use. Google's own canonical offscreen example uses CLIPBOARD
     for exactly this reason. The reason was incorrectly changed to
     AUDIO_PLAYBACK in a prior fix and has been reverted to CLIPBOARD.

  Fix:
  - setupOffscreen now awaits createDocument before calling
    SendSafeTabMessage, guaranteeing the audio.html listener is running
    before FORRHTEMP is sent.
  - SendSafeTabMessage is called once after the if/else block (not
    duplicated in each branch as before).
  - reasons: ['CLIPBOARD'] is intentionally retained — AUDIO_PLAYBACK
    causes Chrome to auto-close the document 30 seconds after audio stops.
  - background_sendOffscreen now awaits setupOffscreen so errors propagate
    rather than being silently swallowed.
  5 new tests.

  FIX — audio.js: onMessage listener "message channel closed" error

  Chrome error: "A listener indicated an asynchronous response by returning
  true, but the message channel closed before a response was received."
  Context: audio.html  Stack: audio.html:0

  Root cause: chrome.runtime.onMessage broadcasts to all extension contexts
  including the offscreen audio.html page itself. The FORRHTEMP handler in
  audio.js sent CONSOLELOG messages via chrome.runtime.sendMessage from
  inside the handler. Those messages looped back into audio.js's own
  listener, hit the if (msg.text == "FORRHTEMP") check, fell through
  without a response, and Chrome fired the channel-closed error for each
  one. Additionally, any other runtime-broadcast message (not FORRHTEMP)
  would cause the listener to return undefined — telling Chrome the channel
  was synchronously closed, which also triggers the error for callers that
  passed a response callback.

  Fix:
  - Added explicit "return false" at the bottom of the onMessage listener
    for all non-FORRHTEMP messages. This tells Chrome the listener will
    not call sendResponse and the channel can be closed immediately.
  - Removed the CONSOLELOG sendMessage calls from inside the FORRHTEMP
    handler. They were the source of the loopback and are pure debug noise
    with no observable in-browser effect (background.js handles CONSOLELOG
    with return false and does nothing with the message).
  - The FORRHTEMP branch continues to return true and call sendResponse
    inside .then()/.catch() — this is unchanged and correct.
  9 new tests.

  FIX — CSS color class specificity: table cell colors invisible

  All colour classes applied to <td> elements (.greentd, .redtdd, .totaltd,
  .bluecalc, .dateWork) were being silently overridden by the base rule:
    .tableClass td { color: var(--text); }
  Specificity of .tableClass td is (0,1,1); bare colour classes are (0,1,0).
  The base rule won on every cell, so green, red, purple, and blue colours
  were never displayed — cells rendered in the default text colour.

  Fix: added qualified rules (.tableClass td.greentd etc.) with specificity
  (0,2,1) above the unqualified fallbacks. Unqualified rules retained for
  non-table uses of the same classes.

  Also fixed in this pass:
  - .tableClass tbody tr:hover td.totaltd added to preserve purple-bg on
    hover (the generic tr:hover td rule washed purple rows to --surface2).
  - .tableClass.cons-view .blueTitle { cursor: default } — pointer cursor
    was stale after Option A disabled row expansion in cons-view.
  - buildTotalCLine Productivity % td given class="bluecalc" for consistency
    with week1/week2/period rows (was unstyled plain text before).
  10 new tests.

  FIX — DM Sans bundled locally instead of remote @import

  popup.css and style.css both loaded DM Sans via:
    @import url('https://fonts.googleapis.com/css2?family=DM+Sans...');

  manifest.json host_permissions has never included fonts.googleapis.com or
  fonts.gstatic.com (the second-hop domain the Google Fonts CSS response
  itself points to for the actual .woff2 files). Extension pages and
  content scripts fetching cross-origin resources are governed by the same
  host-permission model as scripts — without it listed, this remote font
  request was very likely failing silently the entire time. Every popup
  and content-script element styled with 'DM Sans', system-ui, sans-serif
  was actually rendering in the system fallback font, not DM Sans.

  Fix: removed both @import rules and added local @font-face declarations
  (weights 400/500/600) in both files, referencing fonts/*.woff2 by
  relative path. popup.css resolves these against the extension package
  automatically (same-origin, no manifest change needed). style.css is
  declared in manifest.json's content_scripts.css array, so Chrome also
  resolves its relative url() paths against the extension package when
  injecting into host pages — no web_accessible_resources entry needed
  for either file.

  NOTE: The actual DM Sans .woff2 binary files are not yet present in
  fonts/ — see fonts/README.txt for exact filenames and download sources
  (Google Webfonts Helper recommended). Binary font files cannot be
  fetched through this session's available tools; they must be added
  manually before this fix takes full effect in-browser.

  NOTE — Digit alignment in numeric/time columns (deferred)

  DM Sans does not produce consistent digit widths — this woff2 subset has
  no tnum GSUB feature (verified by decompressing the font binary and
  inspecting the GSUB FeatureList with fontTools: only calt/ccmp/dnom/frac/
  liga/locl/numr present; digit advance widths range from 312 to 698 units
  with no tabular alternate). font-variant-numeric: tabular-nums has nothing
  to invoke and is a no-op for this font. A monospace font stack was trialled
  but reverted — the visual size mismatch between monospace and DM Sans was
  unacceptable. Proper fix deferred to the Inter migration (see Roadmap):
  Inter ships tnum by default, at which point tabular-nums will work.

  BUG FIX — audio.js: message channel closed before response

  Chrome error: "A listener indicated an asynchronous response by returning
  true, but the message channel closed before a response was received."

  Root cause: SendSafeTabMessage (utils.js) always passes a callback to
  chrome.tabs.sendMessage, so Chrome expects a response from every listener
  that receives the message. The FORRHTEMP listener in audio.js called
  snd.play() (async) but returned undefined — Chrome closed the channel
  before the promise resolved.

  Fix: added return true to keep the channel open, and added sendResponse
  calls inside the .then() and .catch() handlers so Chrome receives an
  explicit reply once play() settles.

  BUG FIX — Bug 2: worktime select/save/cancel overlapped Surplus and Task
  Description columns

  The three edit controls were inserted via $(appendTo).after(), where
  appendTo was the worktime <td>. In a browser, non-td/th elements inside
  a <tr> are invalid HTML and get ejected from the table into a block box
  before it. MDE's #trackertablediv has overflow:hidden, so the ejected
  controls were clipped; on wider viewports they appeared but pushed into
  adjacent columns. LFH had no body width constraint and no overflow:hidden
  on the table container so the popup expanded to fit.

  Fix — floating panel outside the table:
  - dataclick() captures clickedTd and clickedRow (closest <tr>) before
    creating anything.
  - getBoundingClientRect() on the clicked td and #trackertablediv
    calculates panelTop / panelLeft so the panel lands directly below the
    worktime cell regardless of scroll position.
  - A single #worktime-edit-panel div is appended to #trackertablediv
    (position:absolute, z-index:100) containing the select, Save, and
    Cancel controls — the <tr> is never touched.
  - All five save branches use $(clickedRow).find() instead of
    $(ev.currentTarget).closest("tr") / parentElement traversals.
  - Cancel uses revertTime(clickedRow[0]) instead of
    revertTime(ev.currentTarget.parentElement).
  - removeEditButtons() removes #worktime-edit-panel; legacy id removes
    kept as fallback.
  - scrollIntoView() on #cancel removed — caused the tracker scrollbar to
    jump to top because the absolutely-positioned panel has no scroll
    ancestor. Panel is already positioned correctly via getBoundingClientRect().
  - #trackertablediv gets position:relative in popup.css so the panel
    anchors to it correctly.
  - #worktime-edit-panel styled as a horizontal flex row with border,
    shadow, and DM Sans font.

  BUG FIX — Bug 3: Invoice button (#totals) date column too narrow for labels

  The "Invoice" button triggers buildTable with invoice:true, which appends
  "Week Ending ..." and "Total for the period" summary rows via
  processTotalLine. The standard col 1 (Date) width of 72px is too narrow
  for these labels (~136px at 12px DM Sans).

  Fix — CSS:
  - body.invoice-open { width: auto; min-width: 640px } widens the popup
    while invoice view is active.
  - .tableClass.invoice-view th/td:nth-child(1) sets width:140px /
    min-width:130px with white-space:nowrap.

  Fix — JS in buildTable:
  - Clears invoice-view from #trackerTable and invoice-open from body at
    the top of every buildTable call so they don't persist into normal
    tracker view.
  - When invoice:true, adds invoice-view to #trackerTable and invoice-open
    to body so both the column width and popup width rules take effect.
  - closeInvoice handler removes both classes before triggering ViewTracker
    to rebuild the normal table.
